# Политика доставки и оптимизации изображений

Каноническая политика для растровых изображений и адаптивной разметки в layout-проектах. Связанные темы остаются в соседних правилах — не дублируйте их здесь:

- **Целостность импорта из Figma** (inline SVG, локальные пути, без MCP URL, геометрия вектора): [`figma-asset-integrity.RULE.md`](./figma-asset-integrity.md).
- **`alt` и доступность non-text** (описательный vs декоративный, контраст): [`accessibility-and-w3c.RULE.md`](./accessibility-and-w3c.md).

## Загрузка

- У каждого `<img>` должен быть явный `loading`: по умолчанию `loading="lazy"`, или `loading="eager"` для above-the-fold / LCP-критичных слотов.
- Не опускайте `loading` — выбирайте осознанно для каждого слота.

## Адаптивная доставка

- Используйте `picture` и/или `srcset`, когда отображаемый размер зависит от ширины viewport или DPR.
- Каждый кандидат в `srcset` (или `<source>` в `<picture>`) должен удовлетворять правилу intrinsic sizing **для своего** rendered-размера на breakpoint, где он выбирается.
- Предпочитайте дескрипторы ширины (`w`) и `sizes`, соответствующие реальному CSS layout, чтобы браузер выбирал файл подходящего размера.

## Intrinsic sizing (PageSpeed)

**Основное правило:** пиксельные размеры файла не должны превышать **2×** отображаемый размер на странице.

```
Intrinsic width  ≤ Rendered width  × 2.0
Intrinsic height ≤ Rendered height × 2.0
```

**Пример:** если изображение отображается **600×400** px, экспортируйте/храните не больше **1200×800** px.

### Пороги Lighthouse / PageSpeed Insights

Аудит PSI **«Properly size images»** сообщает об избыточных байтах, когда файлы больше необходимого. Цели для pass:

| Тип доставки | Максимальный избыток относительно идеала для отображаемого размера |
|--------------|----------------------------------------------------------------------|
| Обычный `<img>` (один источник) | **&lt; 4 KiB** избыточных |
| Адаптивный (`srcset`, `<picture>`) | **&lt; 12 KiB** избыточных на кандидат |

### Как привести в соответствие

1. Измерьте или оцените **rendered size** в CSS-пикселях на целевом breakpoint (computed size в DevTools).
2. Вычислите целевой экспорт: **rendered × 2.0** по каждой оси (округление до целых пикселей).
3. Переразмерьте растр до этого целевого размера (или ближайшего стандартного шага вниз, если экспорт из дизайна меньше).
4. Предпочитайте **WebP** вместо JPEG/PNG для фото и сложных иллюстраций (типичная экономия **25–35%** при сопоставимом качестве).
5. Применяйте lossy-сжатие с **quality 80–85**, если brief не требует lossless.
6. Повторно проверьте пути в собранном output под структурированными папками проекта (например `app/img/<feature>/...`).

## Форматы и вес

- По умолчанию **WebP** для растровых фото и иллюстраций, если проект допускает поддержку браузерами.
- Не отгружайте раздутые растры, если есть более лёгкий эквивалент (меньшие размеры, лучший формат или векторный SVG по политике Figma).
- Не апскейлите мелкий экспорт выше порога 2× «ради retina», если rendered-слот меньше — экспортируйте под 2× целевого слота.

## Устойчивость вёрстки

- Вёрстка должна оставаться стабильной при более длинном тексте и при замене изображений с **другими aspect ratio** или размерами в рамках политики 2×.
- Проверяйте отсутствие существенного **CLS**; резервируйте место через `width`/`height` или CSS `aspect-ratio` где уместно.

## Проверка

- Запустите **PageSpeed Insights** (или Lighthouse в Chrome DevTools) и убедитесь, что аудит **«Properly size images»** в **зелёной** зоне **без предупреждений** для затронутых страниц.
- В evidence сдачи ссылайтесь на результат `commands/performance-checklist.md` (оптимизации, статус PSI/Lighthouse, отложенные пункты).
- **Figma-driven:** выполните `commands/validate-figma-assets.md` для путей/целостности; sizing растра — по этому правилу.
- Локальный HTML: `npm run qa` / `npm run validate:html` после build при изменении шаблонов.

## Блокирующие условия

- Задачи с новым или изменённым растром **не завершены**, пока любой слот нарушает **Intrinsic ≤ Rendered × 2.0** или пока PSI **«Properly size images»** показывает fail/warning для этих URL — если в brief нет явного waiver (та же позиция, что waivers mockup fidelity в [`WORKFLOW.md`](../../../.cursor/WORKFLOW.md) §1.2).
