# Паттерны JavaScript-Архитектуры

- По умолчанию используй progressive enhancement: базовый контент и контролы должны оставаться рабочими без custom JS, где это возможно.
- Используй unobtrusive JavaScript: поведение через `data-*` и `js-*` hooks (классы в приоритете; `js-` id при необходимости) — см. [`javascript-minimalism.RULE.md`](javascript-minimalism.RULE.md) § «Именование JS-хуков».
- Для стандартных интерактивных сценариев сначала применяй Flowbite и только потом добавляй custom-логику.

## Functional Core / Imperative Shell

- Преобразования данных и логику принятия решений держи в pure helper-функциях.
- DOM-чтение/запись, подписки на события и side effects держи в тонком imperative shell.
- Не смешивай тяжелую бизнес-логику напрямую в event handlers.

## Lifecycle Инициализации

- Для каждого custom интерактива должен быть idempotent init-путь (безопасный повторный вызов).
- Не допускай повторных bindings/listeners при re-init.
- По возможности ограничивай инициализацию релевантным корневым контейнером.

## Ограничения По Простоте

- Делай модули feature-ориентированными и сфокусированными на одной interaction-задаче.
- В build/config-скриптах и orchestration-коде отдавай приоритет читаемости, а не компактности.
- Тернарный оператор используй только для коротких однострочных выражений.
