# Конвенции HTML и Nunjucks

- Использовать Nunjucks loops/macros/includes для повторяющихся паттернов (карточки, списки, сетки).
- Держать шаблонные фрагменты небольшими и предметными (section, card, nav, media).
- Предпочитать data-driven рендеринг вместо копирования HTML-блоков.
- Поддерживать единообразие именования атрибутов и структуры контента в циклах.
- Добавлять краткие комментарии только для исключений и неочевидных ограничений.
- Если у элемента есть и визуальные классы, и JS hook-классы, размещай `js-*` классы в `class`-атрибуте после всех визуальных классов.
- Для `id` как JS-hook в своём коде — префикс `js-`; `id` без `js-` только для контрактов библиотек в `data-*` — см. [`javascript-minimalism.RULE.md`](./javascript-minimalism.md) § «Именование JS-хуков».
- В utility-heavy разметке держи atomic-классы в каноническом порядке `prettier-plugin-tailwindcss`.
- Соблюдай корректную иерархию заголовков (`h1` -> `h2` -> `h3`) без пропусков уровней в структуре документа.
- Рассматривай `section` и `article` как именованные области: внутри каждого должен быть содержательный заголовок.
- Считай site shell зоной ответственности layout: глобальные `header`/`sidebar` держи в корневом layout и общих partials, а не в шаблонах отдельных страниц.
- Шаблоны страниц держи сфокусированными на контенте страницы: по умолчанию используй `{% extends "njk-layouts/_main.njk" %}` и рендерь только `{% block content %}`.
- Создавай дополнительный layout только при подтвержденном структурном расхождении, а не ради разовой настройки отдельной страницы.
- Язык страницы по умолчанию русский: используй `<html lang="ru">`, если в задаче явно не указан другой язык.
- Для viewport используй только такое объявление: `<meta name="viewport" content="width=device-width, initial-scale=1">`.
- Если финальный SEO-текст не задан, оставляй заглушки мета-тегов для новых страниц:
  - `<meta name="description" content="description">`
  - `<meta name="keywords" content="keywords">`
- В `<title>` указывай название страницы, совпадающее с именем из реестра страниц (`app/index.html`), если явно не запрошено иное.

## Контактные ссылки (email и телефон)

- Если **адрес электронной почты** показан в видимой разметке (footer, блок контактов, подпись автора и т.д.), выводите его как `<a href="mailto:address@example.com">…</a>`. В `href` — реальный адрес (при необходимости URL-encoded); текст ссылки — адрес, если в задаче не указана другая подпись.
- Если **номер телефона** показан в видимой разметке, выводите его как `<a href="tel:…">…</a>`. В `href` — набираемый URI (предпочтительно E.164: `+` и только цифры, например `tel:+74950000000`); видимый текст может сохранять пробелы, скобки и локальное форматирование.
- **Не** оставляйте контактный email или телефон простым текстом в `<li>` / `<p>`, если пользователь должен иметь возможность позвонить или написать.
- Декоративные или неконтактные упоминания (текст статьи, примеры в документации) вне scope, если это не реальные контакты.
- **Проверка:** `rg` по `app/**/*.njk` для контактных блоков с `@` или телефоном — на тех же узлах есть `mailto:` / `tel:`; `npm run build` проходит.

## Link href (заглушки страниц и JS)

- **Реальная навигация:** `href` ведёт на существующую собранную страницу (`*.html`), валидный `mailto:` / `tel:` или якорь `#id` на **той же** странице.
- **Заглушка перехода (цель ещё не свёрстана):** `href="#"`. **Не** придумывайте кросс-страничные hash (`other-page.html#section`) и hash на главной, если в `dist/` нет соответствующего `id`.
- **Якорь только для JS (без навигации):** `href="javascript:;"`. **Не** используйте `href="#"` для чисто поведенческих контролов. Для action-only добавляйте `role="button"` и поддержку клавиатуры (см. [`accessibility-and-w3c.RULE.md`](./accessibility-and-w3c.md)).
- **Запрещено как заглушки:** `href="javascript:void(0)"` без необходимости стороннего сниппета; фиктивные `home-page.html#…`, если фрагмент или страница не реализованы.

**Проверка:**

- В новых/изменённых `<a href=` в `app/**/*.njk`: заглушки → `#`; JS → `javascript:;`; реальные цели → существующие файлы или якоря на той же странице.
- `rg 'href="[^"]*\.html#' app/` — только там, где `id` фрагмента есть на этой странице в `dist/` (иначе заменить на `#`).
- После правок разметки — успешный `npm run build`.

## Shell страницы (footer у низа viewport)

- Корневой layout должен держать глобальный footer **у нижнего края окна**, если контента меньше высоты viewport; при длинной странице footer идёт **после** прокручиваемого контента.
- **Порядок в DOM** в `_main.njk`: `header` (partial) → `main` → `footer` (partial). Не переносите footer внутрь `main`, если бриф не требует иной shell.
- **CSS-контракт:** `body` — колоночный flex-контейнер с минимальной высотой viewport (`min-h-screen` или согласованный эквивалент); `main` — `flex-1` (`grow`), забирает оставшееся пространство между header и footer.
- Реализуйте через `@apply` в CSS проекта (`app/css/components.css`, `@layer base` для `body` и `.main` — см. [`tailwind-usage-policy.md`](./tailwind-usage-policy.md)), без длинных utility-строк на `<body>` в Nunjucks.
- **Не** используйте `position: fixed` / `sticky` на site footer только ради прижатия к низу viewport; sticky header (`site-header`) остаётся отдельно.
- **Проверка:** на короткой странице (`index.html` / пустая `home-page`) в DevTools footer у нижнего края viewport без большой белой полосы снизу; на длинной — высота документа больше viewport, footer после контента. `npm run build` проходит.

## Изображения в шаблонах (media)

Sizing, форматы, `picture`/`srcset` и гейты PageSpeed — в [`image-delivery-and-optimization.RULE.md`](./image-delivery-and-optimization.md). `alt` и non-text a11y — в [`accessibility-and-w3c.RULE.md`](./accessibility-and-w3c.md). Импорт из Figma и целостность SVG — в [`figma-asset-integrity.RULE.md`](./figma-asset-integrity.md).

- У каждого `<img>` в `app/**/*.njk` явно задавайте `loading="lazy"` или `loading="eager"`.
- Когда отображаемый размер известен из layout, задавайте `width` и `height` (или резервируйте место через CSS `aspect-ratio`) на `<img>` для снижения CLS; значения — **rendered** CSS-пиксели, не полный размер исходника.
- Используйте `picture`/`srcset`, если ширина слота меняется по breakpoint; каждый кандидат — в пределах **intrinsic ≤ rendered × 2.0** для этого breakpoint (см. image-delivery rule).
- Растр только через локальные пути проекта (например `app/img/...`); не оставляйте Figma MCP и другие временные remote URL в шаблонах.
- Для повторяющихся слотов (карточки, галереи, аватары) предпочитайте macro или общий partial с единым `src`/`srcset`, `sizes`, `loading`, `width`/`height` и `alt` — не копируйте расходящиеся `<img>` по циклам.
- В data-driven циклах передавайте поля изображения (`src`, при необходимости `srcset`/`sizes`, размеры, `alt`, `loading`) в литералах `{% set %}`; не зашивайте в разметку файлы больше нужного, если macro может выбрать корректный ассет.
- Декоративные изображения: `alt=""` и при уместности `role="presentation"` по accessibility rule; информативные — осмысленный `alt` из данных или текста.

**Проверка:**

- `rg '<img' app/**/*.njk` — у каждого тега есть `loading=` и `alt`; LCP/hero при необходимости с `loading="eager"`.
- Новый или изменённый растр в `app/img/` соответствует image-delivery rule (≤ 2× rendered); выполните `performance-checklist` и убедитесь, что PSI **«Properly size images»** зелёный для затронутых страниц.
- `npm run build` проходит; в `dist/**/*.html` нет пустых `src` и битых путей к изображениям.

## Правила переиспользования

- Если блок встречается более одного раза, выносить его в partial или macro.
- Если меняется только контент, передавать данные вместо клонирования разметки.

## Фикстуры и mock-данные

Гибридная политика: крупные фикстуры **страниц** — в JSON; **partials и mixins** — только `{% set %}`. Это не продакшен-БД.

### JSON (страницы и общие defaults)

- **Крупные фикстуры страницы** (массивы, вложенные объекты, длинные тексты) → co-located `app/<page>.json` рядом с `app/<page>.njk` (одинаковый basename). Gulp подмешивает автоматически, если файл есть.
- **Общие для всех страниц** (`lang`, `siteName` и т.п.) → `app/shared/*.json`. Shallow merge: все `shared/*.json` по алфавиту, затем co-located JSON страницы (страница побеждает при конфликте top-level ключей).
- **Мелкие поля страницы** (1–3 простых значения) — `{% set %}` в начале `.njk` страницы допустим.
- **`{% set %}` на странице перекрывает** одноимённый ключ из JSON.
- **Только shallow merge:** вложенные объекты между `shared/` и page JSON не дополняются — заменяются целиком.
- Не дублируйте один top-level ключ в нескольких `shared/*.json`.
- Порядок слоёв — [`app/shared/README.md`](../../../app/shared/README.md).

### `{% set %}` (partials, mixins, мелкие поля страницы)

- **Partials (`njk-parts/_*.njk`):** mock **только** через `{% set %}` — JSON для partial **запрещён**.
- **Mixins (`_mixins.njk`):** layout-константы **только** через `{% set %}` — JSON **запрещён**.
- **Близость к потреблению (для partial):** данные там, где разметка и циклы.
- **Родитель перед include (допустимо):** `{% set %}` в вызывающем шаблоне **непосредственно перед** `{% include %}`; литералы, не выборка по индексу.
- **Модификатор при include:** `{% include … with { … } %}` для мелких отличий оформления.
- **Для прототипа избегайте:** общий каталог в layout по индексу/ключу; partial только ради `{% set %}`; индексы/`%` для чередования стилей.
- **Область видимости:** `{% set %}` внутри `{% include %}` не экспортируется в родитель.
- **Не** используйте `manageEnv` / `addGlobal` в gulp для фикстур — см. [`gulpfile-universal-starter.RULE.md`](./gulpfile-universal-starter.md).
- **Не** добавляйте partials только ради `{% set %}`.
- **Формат литералов в `{% set %}` (обязательно):** развёрнутая читаемая структура — не однострочные «портянки» объектов.
  - `[` на той же строке, что и `{% set name =`; каждый элемент массива — с новой строки.
  - Каждый объект `{ ... }` — **многострочный**: одна пара ключ–значение на строку, **отступ 2 пробела** на уровень вложенности (массив → объект → вложенный массив).
  - Вложенные массивы (например `tags`) — многострочно, по одному элементу на строку, если элементов два и больше.
  - Запятая после последнего элемента допустима; пустая строка перед `] %}` — только если так читабельнее в больших фикстурах.
  - **Не** сжимайте всю запись в одну строку, если в объекте больше двух полей.
- **Prettier:** парсер Nunjucks/HTML **нельзя** прогонять по многострочным фикстурам `{% set %}` — он схлопывает объекты в нечитаемые переносы. Такие шаблоны — в [`.prettierignore`](../../../.prettierignore). Не запускайте `prettier --write` для ignored-файлов; форматируйте вручную по правилам выше.
- **Проверка:** `gulpData(getTemplateData)` в [`gulpfile.js`](../../../gulpfile.js); нет доменного `require()` фикстур и нет `manageEnv`/`addGlobal`; блоки `{% set %}` соответствуют многострочной схеме; `app/shared/*.json` и co-located `app/*.json` проходят `format:check`; после `npm run build` data-driven блоки в `dist/*.html` не пустые.

## Порог выноса в partial (не дробить без нужды)

- **Не создавайте** новый `app/njk-parts/_*.njk`, если разметка подключается **ровно из одного** родителя и **нет подтверждённого второго потребителя** в scope текущей задачи.
- **Оставляйте разметку inline** в родительском шаблоне, если блок **жёстко связан** с этим родителем (например, модалка поиска только из шапки, одноразовый drawer у одной панели).
- **Выносите в partial или macro**, если выполняется хотя бы одно условие:
  - тот же фрагмент (или будет в ближайшей задаче) подключается через `{% include %}` / import из **двух и более** шаблонов или layout;
  - блок — **переиспользуемая единица** продукта (карточка, хлебные крошки, рекламный слот) с данными/macros на нескольких страницах;
  - родительский файл стал неудобен для навигации **и** фрагмент — **именованная, проверяемая область** (не одноразовая подсекция одного компонента).
- **Не выносите** «ради чистоты»: partial с единственным `{% include %}` в одном файле даёт лишнюю косвенность без выигрыша в переиспользовании.
- **Предпочитайте macro** partial, если паттерн повторяется с разными данными в одном или нескольких файлах, а обёртку оставляете у вызывающего.
- **Shell-partials** (`_header.njk`, `_footer.njk`, layouts) остаются нормой: они в layout и описывают глобальный chrome, а не одноразовые куски.
- **Проверка:** перед добавлением `_something.njk` укажите **точки подключения** (пути файлов). Если точка одна — inline, если в брифе задачи или утверждённом плане явно не требуется отдельный файл (параллельная разработка, генерируемый список include).

## Имена Tailwind-классов в разметке (content scan)

- **Не** собирайте имена Tailwind- или проектных component-классов через подстановку Nunjucks в атрибут `class` (например `site-footer-column-{{ loop.index }}`, `{{ column.modifierClass }}`, `{{ classes[loop.index0] }}`). Сканер Tailwind читает **литеральные** строки в `app/**/*.njk`; динамические фрагменты на этапе сборки не видны, поэтому соответствующие правила `@apply` в SCSS могут быть вырезаны purge, и стили в `dist/css` пропадут без явной ошибки.
- **Делайте** циклы по **данным** (подписи, ссылки, текст); **модификаторы/layout** привязывайте явными литералами, видимыми сканеру:
  - фиксированные ветки `{% if %} / {% elif %}` на каждый известный модификатор (`site-footer-column-1`, `site-footer-column-2`, …);
  - отдельные блоки разметки, если структура различается;
  - или атомарные Tailwind-утилиты в разметке, если набор действительно динамический (см. [`tailwind-usage-policy.md`](./tailwind-usage-policy.md)).
- **Не** закрывайте отсутствие литералов через `safelist` в `tailwind.config.js`, если исправление — в шаблонах; `safelist` — для сторонней или генерируемой разметки вне `app/**/*.njk`.
- Каждый модификатор в разметке сопровождайте именованным правилом в SCSS (`@layer components`, `@apply` по [`tailwind-usage-policy.md`](./tailwind-usage-policy.md)); не заменяйте `@apply` сырым CSS только из-за purge — исправляйте литералы в шаблонах.
- **Проверка:** новые/изменённые модификаторы — **полные литеральные** подстроки в `app/**/*.njk` (`rg 'site-footer-column-[123]' app/**/*.njk` или имена проекта); `npm run build` успешен; в `dist/css/*.css` есть селекторы модификаторов; нет новых `safelist` для классов, которые должны быть видны в шаблонах.
