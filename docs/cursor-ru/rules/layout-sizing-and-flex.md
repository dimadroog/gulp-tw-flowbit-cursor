# Размеры раскладки и flex

> Перевод для человека. **Enforce:** [`.cursor/rules/layout-sizing-and-flex.RULE.md`](../../../.cursor/rules/layout-sizing-and-flex.RULE.md).

Предпочитайте **адаптивные размеры от контента**, а не фиксированные пиксели. Это дополняет [`css-inheritance-layout.RULE.md`](./css-inheritance-layout.md): не наследуемые свойства задаются **на том элементе, которому они нужны**, а не копируются как фиксированные `width`/`height` из экспорта макета, если тот же визуальный результат достижим через flex, padding или поток.

## Избегать явных width и height

- **Не** задавайте фиксированные `width` / `height` / `min-w-*` / `max-w-*` / `min-h-*` / `max-h-*` / `size-*` на обычных layout- и текстовых элементах без веской причины.
- **Допустимо без отдельного обоснования:**
  - **`img`** / **`picture`** / плейсхолдеры видео — intrinsic sizing, резерв под CLS (см. [`image-delivery-and-optimization.RULE.md`](./image-delivery-and-optimization.md), [`html-nunjucks-conventions.RULE.md`](./html-nunjucks-conventions.md)).
  - **Пояс контента** — `.container` / `max-w-content` по [`tailwind-usage-policy.RULE.md`](./tailwind-usage-policy.md).
  - **Иконки** / декоративные ассеты с заданным glyph/hit box, если макет фиксирует размер глифа.
  - **Фиксация пропорций** (`aspect-*`, `aspect-ratio`) для медиа-слотов — предпочтительнее сырой `height`, когда слот ведёт себя как изображение.
- Если при mockup-driven работе нужна размерность, **переводите** пиксели из экспорта в токены, стеки padding, flex-basis или `gap` — не вставляйте `w-[254px] h-[48px]` на каждый соседний узел только потому, что в Figma такие числа (см. **Перевод макета** ниже).

## Интерактивные контролы — высота через padding

Для **кнопок** (`.btn`), **пунктов nav/меню**, **табов** и **нативных полей** (`.form-control`, `.form-check-input`):

- Предпочитайте **`padding`** / **`py-*` `px-*`** (в базовых `@apply` в `components.css` или утилитах в разметке) для вертикального ритма и hit area — **не** фиксированный `h-*`, если в брифе нет жёстко зафиксированной высоты контрола.
- Согласуйте с семантическими базами в [`tailwind-usage-policy.RULE.md`](./tailwind-usage-policy.md) (`.btn` — `inline-flex items-center` + padding, а не один `h-12`).

## Перевод макета (адаптивность вместо копирования)

- **Не** копируйте фиксированные width/height из референса **как есть**, если контент может меняться (длинные подписи, локализация, динамические счётчики).
- Достигайте **визуального совпадения** на контрактных брейкпоинтах ([`WORKFLOW.md`](../WORKFLOW.md) §1.2, [`pixel-perfect-delivery.RULE.md`](./pixel-perfect-delivery.md)) через spacing-токены, flex-распределение и переносы — а не заморозкой каждого бокса в пиксели экспорта.
- Фиксированные размеры из макета — **подсказки** для gap, padding и flex-basis, а не обязательные литеральные `width`/`height` на каждом узле.

## Flex вместо grid

- **По умолчанию:** стройте layout-зоны через **flex** (`flex`, `inline-flex`, `flex-col`, `flex-wrap`, `gap`, утилиты выравнивания), когда структура — одноосный поток или выражается вложенными flex-контейнерами.
- **Не** переходите на **grid** (`grid`, `grid-cols-*`, `grid-rows-*`, `col-span-*`, `row-span-*`), если тот же визуальный и адаптивный результат достижим во flex без хрупких обходных путей.
- **Grid — только когда flex не справляется**, например:
  - настоящий **двумерный шаблон треков** (явные строки **и** колонки, которые должны выравниваться независимо от порядка в разметке),
  - **перекрывающиеся** или именованные **grid areas**, для которых во flex понадобились бы absolute или дублирование разметки,
  - **плотная** плиточная/masonry-подобная раскладка, где span по строкам/колонкам в grid проще и стабильнее, чем вложенный flex с вычисляемыми ширинами.
- Вложенная раскладка: внешняя flex-оболочка и внутренние flex-дети; внутренний grid — **только** на поддереве, которому нужны 2D-треки; не делайте grid по умолчанию для всего page shell.
- Если grid оправдан — зафиксируйте причину одной строкой в CSS-комментарии или отчёте по задаче (что flex не мог выразить).

## Дочерние flex-элементы — сначала flex, потом width

Когда элемент участвует в **flex**-строке/колонке:

1. **По умолчанию:** не задавайте `width` / `min-width` / `max-width`, пока раскладка не ломается без них.
2. **Предпочитайте** `flex-grow`, `flex-shrink` и `flex-basis` (Tailwind: `grow`, `shrink`, `basis-*` или shorthand ниже), чтобы выразить долю пространства.
3. `width` / `min-w-*` / `max-w-*` — только когда flex alone не закрывает макет (например, фиксированный sidebar rail, колонка с truncate и `min-w-0` для overflow).

## Синтаксис flex shorthand (Tailwind)

- **Предпочитайте** произвольный shorthand **`flex-[<grow>_<shrink>_<basis>]`** — напр. `flex-[0_0_254px]`, `flex-[1_1_0%]` — вместо отдельных `grow`, `shrink`, `shrink-0`, `basis-*` на том же элементе.
- Подчёркивания в arbitrary value соответствуют пробелам в CSS (`flex: 0 0 254px`).
- Отдельные `grow` / `shrink-*` / `basis-*` — только когда одна ось или breakpoint-вариант требует другого примитива; где возможно, держите базовый паттерн в одном shorthand-классе.

## Проверка

- [ ] Новая разметка/CSS не добавляет лишних `w-*` / `h-*` / `size-*` на текстовые блоки, flex-детей и контролы без задокументированного исключения.
- [ ] Кнопки, пункты меню, табы и поля формы масштабируются через padding / `.btn` / `.form-control`, а не изолированные фиксированные высоты.
- [ ] Участники flex используют `flex-[*_*_*]` или flex-распределение раньше, чем ограничения по `width`.
- [ ] Пиксельные боксы из макета переведены в адаптивную раскладку, а не вставлены как фиксированные размеры на каждый элемент.
- [ ] Новая раскладка использует flex там, где flex-конструкции достаточно; `grid` / `grid-*` — только где flex не закрывает требование (причина зафиксирована).
