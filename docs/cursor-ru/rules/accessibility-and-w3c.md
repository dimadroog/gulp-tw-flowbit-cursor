# Политика accessibility и W3C

- Соблюдать семантические landmarks и корректную иерархию заголовков.
- Использовать семантические landmarks (`header`, `nav`, `main`, `aside`, `footer`) как обязательный baseline страницы.
- Осторожно использовать `section` и `article`: каждый такой блок должен содержать содержательный заголовок (обычно `h2` или `h3`).
- На каждой странице обязательно должен быть один `h1`. Если по макету нет видимого заголовка, добавляй доступный visually-hidden `h1`.
- Каждый `img` должен иметь атрибут `alt`: осмысленное описание для информативных изображений и пустой `alt=""` для декоративных.
- Все интерактивные элементы должны быть доступны и управляемы с клавиатуры.
- Использовать `:focus-visible` стили с хорошо заметным контрастом.
- Для динамических контролов полностью настраивать `aria-*` (`aria-expanded`, `aria-controls`, labels, states).
- Проверять контраст и не передавать смысл только через фоновые изображения.

## Контраст при отсутствии утверждённого макета

- Если бриф **без финального UI-макета** (нет режима fidelity к Figma/статике по §1.2 `WORKFLOW`), по умолчанию ориентироваться на **WCAG 2.1 уровень AA**: порядка **≥4.5:1** для основного текста и **≥3:1** для крупного текста (ориентир — ≥18 pt / полужирное ≥14 pt) и для **значимых границ/иконок** на смежном фоне.
- Комбинируй **токены палитры** из **`tailwind.config.js`** (**`brand.*`**, **`primary.*`**, **`success|warning|danger`**) поверх **`bg-white`** и других уже заданных поверхностей; не набрасывать «слившиеся» парами пастель/серое на белом без проверки.
- Если пара сомнительна — **проверить** калькулятором контраста (DevTools браузера, WebAIM и т.п.) и **поправить** яркость/оттенок до допустимых значений.
- Сознательно слабый контраст — только при **явном** исключении в задаче по аналогии с waivers в §1.2 `WORKFLOW`.

## Структура документа и `headingoffset`

- На странице **один** `h1` — заголовок верхнего уровня (заголовок страницы или эквивалент). Скринридеры и алгоритмы outline по умолчанию считают **каждый** «голый» `h1` верхнеуровневым, если не задан offset.
- Если блок **демонстрирует** или **включает** разметку статьи/WYSIWYG со своей лестницей `h1`–`h6` (образцы типографики в design-system, превью `.typography`, тело фикстуры), **нельзя** оставлять второй document-level `h1` без коррекции.
- **Обязательно:** на обёртке внутренней лестницы (например `.typography`, демо-панель, цитируемое тело статьи) задай **`headingoffset="1"`**, чтобы внутренний `h1` учитывался на уровень глубже (в outline как `h2`). Увеличивай значение, если демо лежит глубже по секциям.
- **Альтернативы** (только если `headingoffset` не подходит): показывай в демо только `h2`–`h6`, либо несемантические элементы со стилями под заголовки — но сохраняй настоящий `h1` в демо только вместе с `headingoffset`.
- **Проверка:** W3C Nu Checker (или аналог) не предупреждает, что несколько `h1` считаются верхнеуровневыми; после `npm run build` — **`npm run validate:html`** с результатом **pass**.

## Модальные окна (`div` + `aria-labelledby`)

- У `div` неявная роль **`generic`**. **`aria-labelledby`** (и **`aria-label`**) на `div` **недопустимы**, пока не задана явная роль **кроме** `caption`, `code`, `deletion`, `emphasis`, `generic`, `insertion`, `paragraph`, `presentation`, `strong`, `subscript`, `superscript`.
- **Корень Flowbite / кастомной модалки:** на **внешнем** контейнере (узел с `data-modal-target` / `id` для открытия-закрытия) вместе укажи **`role="dialog"`**, **`aria-modal="true"`** и **`aria-labelledby`** (или **`aria-label`**). Не оставляй `aria-labelledby` на внешнем `div` без роли, а `role="dialog"` — только на внутренней карточке.
- **`aria-labelledby`** должен ссылаться на **`id`, существующий в DOM** в открытом состоянии (видимый заголовок или `.sr-only` подпись). Не дублируй один `id` между формой и экраном успеха.
- Не вкладывай **`role="dialog"`** друг в друга.
- **Проверка:** W3C Nu Checker не выдаёт ошибку о запрете `aria-labelledby` на `div`; общие partial модалок проходят **`npm run validate:html`**.

## Чеклист взаимодействия

- Tab order соответствует визуальной и логической структуре.
- Для modal/offcanvas обеспечено корректное управление фокусом и закрытием; корни модалок — по разделу **Модальные окна** выше.
- Декоративные элементы скрыты от assistive technologies, где это необходимо.
