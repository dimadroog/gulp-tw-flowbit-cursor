# add-rule

Используй эту slash-команду, когда нужно **добавить или обновить регламент** для **проектов вёрстки в целом** — переносимый слой политики в `.cursor/` (`rules/*.RULE.md`) или согласованно поправить оркестрацию (`commands/*.md`, `WORKFLOW.md`).

Это **не** разовые заметки про текущий экземпляр репозитория (бриф клиента, контент страниц, waivers спринта). Рабочий репозиторий — копия общего `.cursor`-пакета; правила здесь должны оставаться валидными при переносе пакета в другой layout-проект.

Каноническая карта: [`WORKFLOW.md`](../WORKFLOW.md) (**§4** — rules vs commands vs skills). Решения уровня конкретного проекта — в выводе [`init-layout-project.md`](init-layout-project.md) или в документации проекта, а не в устойчивых rules, пока их не обобщили.

## Обязательства (fail-fast)

1. **Пойми каркас** — прочитай [`WORKFLOW.md`](../WORKFLOW.md) в рамках вызова (таблица жизненного цикла, §4 классификация, двуязычное правило для `.cursor/`).
2. **Просмотри существующие политики** — перечисли `rules/*.RULE.md`, пробеги [`rules/workflow-orchestrator.RULE.md`](../rules/workflow-orchestrator.RULE.md) и [`commands/run-layout-task.md`](run-layout-task.md) **до** предложения места правки — чтобы навигируемость не страдала (одна тема на правило; не дублируй список стека или цепочек гейтов, если явно не обновляется `WORKFLOW`).
3. **Сформулируй запрос пользователя** как **одну исполнимую политику**:
   - кто применяет (агент / инструменты),
   - когда срабатывает (всегда, тип задачи, охват файлов),
   - что запрещено / обязательно и **как проверить** («как понять, что сделано»).
   - формулируй так, чтобы политика была осмысленна для **другого** layout-репозитория с тем же `.cursor`-пакетом (без «этот сайт/проект/клиент»).
4. **Переносимый охват governance** — перед записью убедись, что политика относится к `rules/*.RULE.md`:
   - **В scope:** повторяющиеся практики вёрстки (HTML/Nunjucks, Tailwind, Flowbite, gulp/`npm run qa` по `WORKFLOW` §1.1–2), устойчивые пути как **конвенции класса** (`app/`, `dist/`, `njk-layouts`, partials layout-shell), абстрактные зоны UI (шапка, hero, ключевая карточка).
   - **Вне scope для rules:** имена клиента/бренда, домены, URL конкретных страниц, один файл Figma или одно имя страницы как норма, ID спринта/тикета, waivers итерации («пока в этом репо…»).
   - **Если запрос instance-specific:** фиксируй в файлах проекта (`PROJECT_PROGRESS.md`, design-system, бриф задачи) или в чеклисте `init-layout-project`; **не** создавай `.RULE.md`. Если пользователь настаивает на rule — обобщи формулировку и укажи в отчёте `redirected` / `generalized`.
5. **Барьер неоднозначности** — если не ясны охват, применимость, строгость (блокирующее vs рекомендация), целевые файлы или **переносимая политика vs запись только для репо**, задай **минимум** уточняющих вопросов (например: «политика для всех layout-проектов или разовая запись для этого репозитория?») и **не пиши** правки, пока не прояснено.
6. **Размещение**
   - **Слияние** — если тему уже ведёт существующий `.RULE.md`, добавь лаконичный пункт или подраздел; при необходимости убери дубли в других файлах.
   - **Новое правило** — если тема новая, конфликтует с рамкой другой политики или раздувает существующую правку **примерно свыше ~500 строк**, создай `rules/<kebab-тема>.RULE.md` с YAML frontmatter как у соседних (`description`, `alwaysApply`, при необходимости `globs` — копируй принятый в проекте формат).
7. **Правки оркестрации** — если требование меняет **цепочки гейтов**, типы задач или шаги `npm`/QA, сначала правь **канон** ([`WORKFLOW.md`](../WORKFLOW.md)), затем [`commands/run-layout-task.md`](run-layout-task.md) и другие задействованные команды; не прячь единичные пункты в нерелевантных правилах.
8. **Имена и навигация**
   - Файлы правил — только `*.RULE.md` в `rules/`.
   - Имя файла = устойчивый slug темы (kebab-case), не спринт и не тикет.
   - Поле `description:` в frontmatter должно читаться как поисковая строка для будущих сессий.
9. **Двуязычный lockstep** — любое изменение под `.cursor/` требует **`sync-cursor-bilingual-structure`**: сразу зеркало в [`.cursor/_RU/`](../_RU/) (английский источник, русское зеркало; §6 `WORKFLOW`).

**Антипаттерн (перепиши перед merge):**

- Плохо: «На `index.njk` CTA должна быть `bg-blue-600`.»
- Хорошо: «Primary CTA в критичных зонах используют токен primary проекта, а не произвольный hue Tailwind.»

## Формат отчёта

Краткий отчёт:

| Пункт | Заметки |
|--------|---------|
| Требование (финал) | Согласованная формулировка политики одним абзацем |
| Переносимость | `pass` — обобщено; `redirected` — вынесено из rules; `fail` — остались instance-specific формулировки |
| Размещение | куда влито / новые имена файлов; затронут ли `WORKFLOW`/оркестрация |
| Изменённые файлы | список + одна строка обоснования на файл |
| Открытые вопросы | нет, или что осталось у пользователя |

Не объявляй завершение, пока явно не указан статус зеркала (**сделано / N/A**) и полный список файлов.
