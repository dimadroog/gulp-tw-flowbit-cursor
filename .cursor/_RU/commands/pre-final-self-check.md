# pre-final-self-check

Используй эту короткую самопроверку непосредственно перед финальным ответом в задачах реализации.

## Обязательные yes/no-проверки

1. Есть ли незакрытые визуальные TODO в mockup-driven задаче? (`yes` блокирует завершение)
2. Запущены ли и явно отражены все применимые blocking-gates? (`no` блокирует завершение)
3. Избежаны ли формулировки "позже/следующим шагом" для блокирующих работ? (`no` блокирует завершение)
4. Для Figma-driven задач проверена ли целостность ассетов (`validate-figma-assets`)? (`no` блокирует завершение)
5. Есть ли явный статус `pass|fail` по pixel-perfect там, где это применимо? (`no` блокирует завершение)
6. Есть ли evidence (команда/проверка) по каждому применимому gate? (`no` блокирует завершение)
7. Для задач с HTML-выводом явно ли `pass` у **обоих** `validate-html` и `validate:w3c` (или полный `npm run qa` после сборки без падения HTML-gates)? (`no` блокирует завершение)

## Правило принятия решения

- Если срабатывает любой блокирующий ответ, задачу нельзя помечать завершенной.
- Нужно продолжить реализацию до устранения блокеров и затем повторить эту проверку.

## Формат результата

- Короткий чеклист:
  - пункт
  - yes/no
  - флаг block
- итоговое решение: `ready_for_finalize | blocked`
