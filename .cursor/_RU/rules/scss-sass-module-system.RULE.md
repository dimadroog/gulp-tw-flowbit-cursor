# SCSS и модули Sass

- В проектных `.scss` **не использовать** Sass **`@import`** (в Dart Sass помечен deprecated). Использовать только **`@use`** и **`@forward`**.
- **`@use "partial"`** (или **`@use "partial" as *`** только при явном намерении дать членам глобальную доступность без namespace) для загрузки partials и переменных/mixins/functions.
- **`@forward`** — для barrel-partials, реэкспортирующих API; не возвращаться к паттернам глобального засорения, характерным для `@import`.
- В **entry**-файлах Sass, которые собирает gulp (`app/scss/*.scss`), **все `@use` и `@forward` — только перед любыми директивами `@tailwind` и прочим выводимым CSS** — требование языка Sass. У проекта: типографика в `_typography.scss` в блоке `@layer base { … }`, подключение через `@use` в начале `style.scss` до `@tailwind`; слои корректно сходятся в postcss-цепочке Tailwind (`app/scss/style.scss`).
- **CSS-переменные (`--*`):** не оставлять в авторском SCSS **неиспользуемых** токенов: если написано `var(--foo, запасное)`, но `--foo` нигде не задаётся (`:root`, родительский компонент, инлайн в шаблоне) — **убрать** переменную и оставить конкретное значение (или явно задать `--foo` там, где она реально выставляется). Не вводить «заглушечные» свойства без сеттера.
- **Проверка:** `rg '@import\\b' app/scss` без совпадений в авторском SCSS; `npm run build` / `gulp build` без предупреждений Sass про deprecated `@import` на файлах проекта; при поиске `var(--` не должно оставаться мёртвых переменных без определения.
